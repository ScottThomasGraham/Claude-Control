# Research: IronRDP as the RDP engine

*Captured 2026-05-30. Sourced web research synthesized into a brief.*

IronRDP is Devolutions' pure-Rust RDP implementation (co-authored with Teleport). It is a Cargo
workspace with a meta crate that re-exports feature-gated sub-crates.
([repo](https://github.com/Devolutions/IronRDP),
[architecture](https://github.com/Devolutions/IronRDP/blob/master/ARCHITECTURE.md))

## 1. Crate layout & how to embed
Depend on the `ironrdp` meta crate and enable features; each feature maps to a sub-crate re-exported
as `ironrdp::<name>` ([deepwiki](https://deepwiki.com/Devolutions/IronRDP)):
- **ironrdp-pdu** — wire structs/codec (X.224, MCS, FastPath). **ironrdp-connector** — client connect
  state machine (CredSSP, license exchange). **ironrdp-session** — `ActiveStage`, fast-path
  processing, SVC/DVC dispatch. **ironrdp-graphics** — bitmap/RemoteFX decode, color conversion,
  `DecodedImage`. **ironrdp-input** — builds FastPathInput packets. **ironrdp-cliprdr / -rdpdr /
  -dvc** — clipboard, device redirect, dynamic-channel multiplexing. **ironrdp-acceptor / -server** —
  server side. **ironrdp-tokio / -blocking** — async vs sync I/O adapters. **ironrdp-tls** —
  rustls/native-tls upgrade. **ironrdp-web** — WASM.

For a headless client: `connector + session + graphics + input + tls + tokio` (use `ironrdp-tokio`;
the official example uses `ironrdp-blocking`). The canonical embedding reference is
**`crates/ironrdp/examples/screenshot.rs`**
([source](https://github.com/Devolutions/IronRDP/blob/master/crates/ironrdp/examples/screenshot.rs))
— it connects, decodes graphics, and writes a BMP. The `ironrdp-client` binary is the full SDL
reference client.

## 2. Graphics / codecs
Mature client-side decode: **interleaved/planar bitmap (RLE)** and **RemoteFX** are implemented in
`ironrdp-graphics`. Decoded output is exposed as a **`DecodedImage`** in **`PixelFormat::RgbA32`**
(RGBA), trivially convertible to an `image::ImageBuffer<Rgba<u8>>` — i.e. you get a screenshot
surface directly. **EGFX/GFX** has working infrastructure (ZGFX decompression, WireToSurface
dispatch, frame StartFrame/EndFrame + FrameAcknowledge) and **H.264 AVC420/AVC444** is the primary
path exercised. **Gaps:** ClearCodec and **RemoteFX Progressive** encode/decode are not implemented,
and much EGFX maturity work to date is server-side ([issue #1158](https://github.com/Devolutions/IronRDP/issues/1158)).
For a controller you can negotiate down to RemoteFX/bitmap reliably; full GFX-everything is not
guaranteed.

## 3. Input injection
`ironrdp-input` provides a **`Database`** (in-memory keyboard/mouse state) and an **`Operation`**
enum; you push operations and `apply()` produces **FastPathInput** PDUs to write to the stream.
Supporting types: **`MousePosition`** (absolute move), **`MouseButton`**, **`WheelRotations`**,
**`Scancode`** (PS/2 Set 1), plus `KeyboardState`/`MouseButtonsState`. Unicode key entry and wheel
are supported ([docs.rs/ironrdp-input](https://docs.rs/ironrdp-input/latest/ironrdp_input/)).

## 4. Auth / security
**NLA/CredSSP is supported** (via the Devolutions `sspi` crate, NTLM/Kerberos) and is integrated into
`ironrdp-connector` — set `enable_credssp: true` in `connector::Config`. **TLS** uses rustls or
native-tls via `ironrdp-tls`; the server public key is extracted from the peer X.509 cert during
`tls_upgrade()`. This is exactly what a default Win10/11 host requires (NLA-on). Note: cert
validation in the examples is permissive (key pinning/trust handled by you).

## 5. DVC / CLIPRDR / RDPDR
**DVC is supported client-side**: `ironrdp-dvc` implements the DRDYNVC channel and traits to register
custom dynamic virtual channels — directly relevant to a Windows-side helper agent talking back over
a private DVC. **CLIPRDR** (`ironrdp-cliprdr`) and **RDPDR** (`ironrdp-rdpdr`) are implemented.

## 6. Maturity, license, production
**Dual MIT OR Apache-2.0.** In production: it powers the **Devolutions Gateway web client /
iron-remote-desktop** (the WASM build) — direct RDP, no intermediate protocol. Active development.
**Caveats:** ClearCodec + RemoteFX Progressive missing; EGFX maturity skewed server-side; audio
(`rdpsnd`) and some redirections are partial.

## 7. Minimal connect+frame+input sketch (tokio)
```rust
let cfg = connector::Config { credentials: Credentials::UsernamePassword{..},
    desktop_size: DesktopSize{width:1280,height:1024}, enable_tls:true,
    enable_credssp:true, .. };
let stream = TcpStream::connect(addr).await?;
let mut framed = ironrdp_tokio::TokioFramed::new(stream);
let connector = connector::ClientConnector::new(cfg, client_addr);
let should_upgrade = ironrdp_tokio::connect_begin(&mut framed, &mut connector).await?;
let (upgraded, server_pubkey) = tls_upgrade(framed, server_name).await?; // rustls
let result = ironrdp_tokio::connect_finalize(upgraded, &mut framed,
    connector, server_name, server_pubkey, &mut NetworkClient).await?;
let mut image = DecodedImage::new(PixelFormat::RgbA32, w, h);
let mut active = ActiveStage::new(result);
let mut input_db = ironrdp_input::Database::new();
loop {
    let (action, payload) = framed.read_pdu().await?;
    for out in active.process(&mut image, action, &payload)? {
        match out {
            ActiveStageOutput::ResponseFrame(f) => framed.write_all(&f).await?,
            ActiveStageOutput::GraphicsUpdate(_) => { /* image updated -> screenshot */ },
            ActiveStageOutput::Terminate(_) => return Ok(()),
            _ => {}
        }
    }

}
```

## Bottom line
**IronRDP is a solid foundation** for an agent-driven headless controller: decoded RGBA framebuffer
surfaces for screenshots, a structured input API for absolute mouse/keyboard/wheel, NLA/CredSSP that
satisfies default Win10/11, and client-side DVC for a custom Windows helper — permissive license,
proven in Devolutions' production web client. **Risks:** uneven graphics codec coverage (plan to
negotiate RemoteFX/bitmap, expect H.264 as the GFX path); cert-trust policy and input-PDU plumbing
are ours to own; pre-1.0 async APIs may shift. None are blockers.
