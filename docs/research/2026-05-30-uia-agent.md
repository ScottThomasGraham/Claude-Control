# Research: Windows UIA helper agent (the semantic upgrade)

*Captured 2026-05-30. Sourced web research synthesized into a brief.*

## 1. UI Automation API basics
UIA presents the desktop as a tree of `IUIAutomationElement` objects. A client `CoCreateInstance`s
`CUIAutomation` → `IUIAutomation`, calls `GetRootElement()`, then walks with an
`IUIAutomationTreeWalker` (use `ControlViewWalker` to skip noise) — or does one `FindAll` with a
`TrueCondition` + a `CacheRequest` to bulk-fetch properties in a single cross-process round-trip
(far faster than per-element COM calls). Per element: `CurrentControlType`, `CurrentName`,
`CurrentBoundingRectangle` (`RECT`).
([walk the tree](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-howto-walk-uiautomation-tree),
[property IDs](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-automation-element-propids))

Crucially, **`BoundingRectangle` returns absolute *physical* screen coordinates** — directly usable
for click targeting. DPI caveat: UIA always reports physical pixels, so a DPI-*unaware* process gets
virtualized/scaled values that won't match reality on >96 DPI or multi-monitor. **Make the agent
per-monitor DPI-aware** (`PerMonitorV2` manifest or `SetProcessDpiAwarenessContext`).
([UIA & screen scaling](https://learn.microsoft.com/en-us/dotnet/framework/ui-automation/ui-automation-and-screen-scaling))

## 2. Language/toolkit
- **C# (.NET)** — fastest to write; `System.Windows.Automation` is older/slower; single distributable
  means AOT or a fat self-contained publish.
- **C++/COM** — smallest binary, most boilerplate.
- **Rust via `windows`/windows-rs** — exposes the full UIA surface
  (`IUIAutomation`..`IUIAutomation6`, `IUIAutomationElement`, `IUIAutomationTreeWalker`) under
  `windows::Win32::UI::Accessibility`, gated by the `Win32_UI_Accessibility` feature.
  ([IUIAutomation docs](https://microsoft.github.io/windows-docs-rs/doc/windows/Win32/UI/Accessibility/struct.IUIAutomation.html))
  The higher-level [`uiautomation`](https://crates.io/crates/uiautomation) crate wraps it ergonomically.
  Yields a **single ~1–3 MB static `.exe`, no runtime dependency**, plus easy TCP/JSON.

**Recommendation: Rust + windows-rs** (optionally the `uiautomation` crate) — best matches "one small
self-contained binary." C# (NativeAOT) is the fallback if the team is .NET-native.

## 3. Session/interactivity constraint
Services run in **Session 0**, isolated from interactive desktops — a service can never read the live
UI directly. Patterns:
- SYSTEM bootstrapper: `WTSGetActiveConsoleSessionId` → `WTSQueryUserToken` → `CreateProcessAsUser`
  into the interactive session.
- **Cleanest/most reliable: a Scheduled Task with an *At log on* trigger, user context, "Run with
  highest privileges."** Task Scheduler places the process in the interactive session and handles
  RDP-reconnect lifecycle — no token plumbing.
([launch interactive from service](https://learn.microsoft.com/en-us/archive/blogs/winsdk/launching-an-interactive-process-from-windows-service-in-windows-vista-and-later),
[security contexts for tasks](https://learn.microsoft.com/en-us/windows/win32/taskschd/security-contexts-for-running-tasks))

## 4. Transport back to controller
- **(a) localhost TCP + existing SSH tunnel** — agent binds `127.0.0.1:PORT`; controller reaches it
  through the SSH port-forward it already has. No firewall exposure, no extra auth surface, trivial.
- **(b) RDP Dynamic Virtual Channel** — doable via `WTSVirtualChannelOpenEx` +
  `WTS_CHANNEL_OPTION_DYNAMIC`, but requires a paired client-side COM plugin (`IWTSPlugin`) in mstsc,
  registry registration, and channel lifecycle handling. High effort for marginal benefit here.
- **(c) plain network TCP** — exposes a port, needs its own auth/TLS; avoid.

**Recommendation: (a) localhost TCP over the existing SSH tunnel** — simplest, loopback-only,
reuses infrastructure we already operate. Newline-delimited JSON snapshots of
`{controlType, name, rect}`.

## 5. Auto-deployment / rollout
Push over SSH (sftp the binary to `%ProgramData%\ClaudeControl\agent-<version>.exe`), then run an
**idempotent PowerShell installer**: check for an existing task (`Get-ScheduledTask`), compare an
embedded `--version`, re-register only if newer. Register with `Register-ScheduledTask` (logon
trigger, user principal, highest privileges) or `schtasks /Create /SC ONLOGON /RL HIGHEST /F`. Start
immediately. **Uninstall:** `Unregister-ScheduledTask -Confirm:$false`, kill the process, delete the
directory. Keep all state under one directory; version directories make rollback trivial.

## 6. Security/permissions
Reading the UIA tree of your own session needs no admin. Admin/UAC matters for: registering a task
with **highest privileges** (so the agent can read *elevated* apps' UIs — UIA can't cross into a
higher integrity level otherwise) and writing under `%ProgramData%`. So the *installer* runs
elevated; the *agent* runs at the user's level (highest available token). Unsigned exes draw
**SmartScreen/Defender** "unknown publisher" friction; **Authenticode signing** ties the binary to a
known publisher and builds reputation. (EV no longer instantly bypasses SmartScreen.)

## Bottom line
Build the agent in **Rust with windows-rs** (`Win32_UI_Accessibility`) for a single small
dependency-free `.exe` that walks the UIA `ControlViewWalker` with a cached `FindAll` and emits
`ControlType/Name/physical BoundingRectangle` as JSON — **per-monitor DPI-aware** so coordinates are
click-ready. Keep it always-available via a **Scheduled Task, At-logon trigger, user context,
highest privileges** (sidesteps Session-0 isolation with zero token plumbing). Ship snapshots over a
**loopback TCP socket reached through the existing SSH tunnel**. Install/upgrade/uninstall with an
idempotent elevated PowerShell script keyed on an embedded version; sign the binary to reduce
SmartScreen/AV friction.
